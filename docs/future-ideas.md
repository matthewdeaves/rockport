# Future Ideas

## Additional rate limiting at the Cloudflare edge

Cloudflare Rate Limiting rules could throttle requests before they reach the tunnel, protecting against brute-force key guessing or abuse from a single IP. The current setup relies on LiteLLM's per-key rate limits (60 RPM / 200K TPM), which only apply after the request reaches the proxy. Edge rate limiting would reduce load on the instance during an attack.

## SNS notifications on instance stop

When the idle-stop Lambda stops the instance, it could publish to an SNS topic to notify the operator. Currently stops are visible in CloudWatch but don't trigger a notification. Useful if you want a push alert when the instance goes idle.

## Pipeline Orchestration (Canvas-to-Reel in One Call)

A `POST /v1/videos/pipeline` endpoint that accepts a character description, reference image, and list of shots, then orchestrates: (1) Nova Canvas IMAGE_VARIATION per shot with consistent seed/cfgScale to generate per-shot frames, (2) Nova Reel MULTI_SHOT_MANUAL with the generated frames. Eliminates the multi-step manual workflow. Deferred because it's the most complex and opinionated feature — better to prove the building blocks first.

## Nova Lite Prompt Rewriting

Opt-in parameter on `/v1/videos/generations` that calls Nova Lite (LLM) to convert rough animation intent into an optimised Nova Reel prompt before submission. Follows the AWS storyboarding pipeline pattern. Excluded because it adds latency (~2-5s), cost (LLM invocation), and an LLM dependency to what should be a video endpoint.

## Cloudflare Access with identity provider

The current setup uses a service token (machine-to-machine). For broader team access, you could add an identity provider (e.g. GitHub OAuth, Google) so team members authenticate via browser before accessing the proxy. Not needed for the current single-operator setup but useful if sharing access more broadly.

## Claude Code Hooks

Hooks that could improve safety, quality, and workflow when using Claude Code with this project.

| # | Hook | Event & Matcher | Type | What it does | Why it matters for Rockport | How it works | Why this type? |
|---|------|----------------|------|-------------|---------------------------|-------------|----------------|
| 1 | Protect sensitive files | PreToolUse → Edit\|Write | command (deterministic) | Blocks edits to `.env`, `.tfvars`, `requirements.lock` by checking file path against a hardcoded list | These files contain Cloudflare API tokens, real infra config, and pinned dependency hashes. Accidental edits leak secrets or break supply chain integrity | Script reads `tool_input.file_path` from stdin JSON, matches against protected patterns, exits 2 to block | Security boundaries should be absolute. Regex on a known file list is faster, cheaper, and has zero false negatives. An LLM might reason itself into allowing an edit |
| 2 | Detect secrets in edits | PreToolUse → Edit\|Write | prompt (LLM) | Scans the content of every edit for things that look like API keys, tokens, passwords, or credentials being hardcoded | Rockport manages AWS creds, Cloudflare tokens, SSM parameters, and LiteLLM API keys. Easy to accidentally paste one into a config file | Haiku receives the `tool_input` (file path + content) and answers ok:false if it sees secret-shaped strings | Secrets have diverse formats — AWS keys, UUIDs, base64 tokens, JWTs. Regex can't catch them all without massive false positive rates. LLM pattern recognition handles the variety |
| 3 | Block dangerous commands | PreToolUse → Bash | prompt (LLM) | Reviews every bash command for destructive intent — terraform destroy, force push, SSM parameter deletion, etc. | Rockport manages real AWS infrastructure and Cloudflare tunnels. A careless `terraform destroy` takes down the whole proxy | Haiku receives the `tool_input.command` and judges whether it's destructive to infra, git history, or credentials | Commands can be obfuscated: `cd terraform && destroy`, `bash -c "..."`, piped chains. Regex is trivially bypassed. LLM understands intent regardless of how the command is structured |
| 4 | Auto-format Terraform | PostToolUse → Edit\|Write | command (deterministic) | Runs `terraform fmt` on any edited `.tf` file automatically | CI enforces `terraform fmt -check` — catching format issues locally avoids a failed CI round-trip. **Implemented in `.claude/hooks/auto-format.sh`** | Script reads `tool_input.file_path`, checks if it ends in `.tf`, runs `terraform fmt` on it if so | `terraform fmt` is idempotent and deterministic by design. An LLM adds nothing — the canonical tool is both faster and definitionally correct |
| 5 | Auto-ShellCheck | PostToolUse → Edit\|Write | command (deterministic) | Runs `shellcheck` on any edited `.sh` file and feeds warnings back to Claude | Rockport has critical bash scripts (bootstrap.sh, rockport.sh) where bugs can brick the EC2 instance. CI runs shellcheck already. **Implemented in `.claude/hooks/auto-format.sh`** | Script reads `tool_input.file_path`, checks if it ends in `.sh`, runs `shellcheck` and returns output as feedback | ShellCheck has fixed, well-defined rules. It's a static analysis tool — asking an LLM to lint bash would be slower and less accurate than the purpose-built tool |
| 6 | Validate Terraform on stop | Stop | agent (LLM + tools) | When Claude finishes a response, runs `terraform validate` and provides targeted feedback if it fails | A syntax error in `.tf` files isn't caught until you run plan/apply. Catching it immediately keeps the feedback loop tight | Agent runs `terraform validate`, reads the error, checks which files Claude modified via git diff, and explains the specific issue | Raw `terraform validate` errors are cryptic. An agent can cross-reference the error with the actual diff and say "you removed `var.x` from variables.tf but main.tf still references it" — actionable vs opaque |
| 7 | Smart post-compaction context | SessionStart → compact | agent (LLM + tools) | After context compaction, reads git diff to see what's being worked on, then injects only the relevant CLAUDE.md constraints | CLAUDE.md has 100+ lines of constraints. After compaction, all are lost. A static reminder dumps everything, wasting context. Targeted injection keeps only what matters | Agent reads `git diff`, reads CLAUDE.md, selects entries relevant to the files being touched | Deciding which constraints are relevant requires understanding both the constraints and the work in progress. A deterministic approach would need a manually maintained file-to-constraint mapping that drifts. An LLM reads both and matches naturally |
| 8 | Desktop notifications | Notification | command (deterministic) | Fires `notify-send` when Claude is waiting for input | Terraform plan/apply and long deploys mean you tab away. Without a notification you waste time checking back manually | One-liner: `notify-send 'Claude Code' 'Needs your attention'` | There is nothing to reason about. A notification is a side effect, not a decision |
| 9 | Audit log | PostToolUse → Bash | command (deterministic) | Appends every bash command with timestamp to a log file | Rockport manages real AWS/Cloudflare resources. Having a record of what commands were run and when is useful for debugging and incident review | Script reads `tool_input.command` from stdin JSON, appends to log file with timestamp | Logging is append-only and unconditional. An LLM could classify commands by risk level, but the added cost per command isn't worth the marginal value over a plain grep-able log |
