# Research: Multi-Model Video Generation

## R1: S3 Cross-Region Constraint

**Decision**: Bedrock async invoke requires the S3 output bucket to be in the same region as the model. Ray2 (us-west-2) needs its own bucket.

**Rationale**: AWS docs and troubleshooting guidance explicitly state "bucket exists in the same region as your Bedrock service." Confirmed via Luma AI Bedrock documentation.

**Alternatives considered**: Single shared bucket (rejected — would require migrating Nova Reel to us-west-2, and Nova Reel may have the same same-region constraint for us-east-1).

## R2: Luma Ray2 API Format

**Decision**: Ray2 uses a different modelInput format from Nova Reel.

**Key differences**:
- Text-to-video: `{prompt, aspect_ratio, duration, resolution, loop}` (duration is string "5s"/"9s")
- Image-to-video: images passed via `keyframes.frame0` / `keyframes.frame1` with `{type: "image", source: {type: "base64", media_type: "image/jpeg", data: "<b64>"}}`
- Nova Reel: images via `textToVideoParams.images[{format, source: {bytes}}]`

**Rationale**: Each model has vendor-specific Bedrock input schemas. The sidecar must build model-specific payloads while keeping the user-facing API unified.

## R3: Ray2 Pricing Model

**Decision**: Ray2 pricing is resolution-dependent: $0.75/s at 540p, $1.50/s at 720p.

**Rationale**: AWS Bedrock pricing page. This is 9-19x more expensive than Nova Reel ($0.08/s). Cost must be calculated per-model and per-resolution for accurate budget enforcement.

**Alternatives considered**: Flat per-video pricing (rejected — AWS charges per-second based on resolution).

## R4: Ray2 Image Constraints

**Decision**: Ray2 accepts images 512x512 to 4096x4096, max 25MB. Nova Reel requires exactly 1280x720, max 10MB.

**Rationale**: Different models, different constraints. The sidecar needs per-model image validation. Pillow MAX_IMAGE_PIXELS limit must be raised to accommodate Ray2's 4096x4096 max.

## R5: Multi-Region Boto3 Clients

**Decision**: Create one bedrock-runtime client per region at startup (us-east-1 for Nova Reel, us-west-2 for Ray2). Similarly, one S3 client per region for presigned URL generation.

**Rationale**: Boto3 clients are bound to a single region. The existing single-client pattern must become a dict of clients keyed by region. Clients are created once at startup (lifespan), not per-request.

**Alternatives considered**: Single client with region override per-call (rejected — boto3 doesn't support per-call region override).

## R6: Database Schema Migration

**Decision**: Add a `model` column (VARCHAR(30), NOT NULL, DEFAULT 'nova-reel') to `rockport_video_jobs`. Use ALTER TABLE with DEFAULT so existing rows get backfilled automatically.

**Rationale**: The model field is needed in all API responses and for per-model cost calculation. DEFAULT ensures backward compatibility with existing data.

## R7: Presigned URL Generation for us-west-2

**Decision**: The sidecar needs a second S3 client for us-west-2 to generate presigned URLs for Ray2 output. The polling endpoint must use the correct S3 client based on the job's model/region.

**Rationale**: Presigned URLs must be generated by an S3 client configured for the bucket's region. Using the wrong region produces invalid signatures.
